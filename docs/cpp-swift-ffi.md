# C++ / Swift FFI — Lessons Learned

## Use a C Bridge, Not Direct C++ Interop

Swift cannot call C++ directly without the experimental `-enable-experimental-cxx-interop`
flag (fragile, not production-ready for complex class hierarchies). The correct
pattern for production use is a **flat C API** implemented in Objective-C++.

```
C++ engine  →  .mm C bridge  →  magentart_bridge.h  →  Swift
```

The C header uses only C-compatible types (`void*`, `bool`, `float`, `uint32_t`).
Swift imports it via the bridging header (`-import-objc-header`).

## The Bridge File Must Be `.mm`, Not `.cpp`

Even if your bridge implementation has no Objective-C syntax, it **must** use
the `.mm` extension. MLX's inference loop requires an Objective-C autorelease
pool per iteration (handled internally via `magentart::detail::AutoreleasePool`).
Compiling as `.cpp` omits the ObjC runtime linkage and crashes at the first
autorelease call.

```
bridge/
├── include/
│   └── magentart_bridge.h   ← C header, extern "C" block, no C++ types
└── magentart_bridge.mm      ← ObjC++ implementation
```

## Opaque Handle Pattern

Never expose C++ types in the C header. Use a `void*` opaque handle:

```c
typedef void* MagentaEngineRef;
MagentaEngineRef magentart_create(void);
void magentart_destroy(MagentaEngineRef engine);
```

In the `.mm` implementation, cast back with a static cast:

```objc
static inline RealtimeRunner* cast(MagentaEngineRef ref) {
    return static_cast<RealtimeRunner*>(ref);
}
```

Always null-guard every function — Swift can pass a nil `UnsafeMutableRawPointer`.

## Cross-Actor Safety of the Handle

`@MainActor`-isolated Swift code cannot pass `MagentaEngineRef` to the audio
render block (a non-isolated closure) without a compiler error. Wrap it in an
`@unchecked Sendable` class:

```swift
private final class EngineHandle: @unchecked Sendable {
    let ref: MagentaEngineRef
    init?() {
        guard let r = magentart_create() else { return nil }
        ref = r
    }
    deinit { magentart_stop(ref); magentart_destroy(ref) }
}
```

This is safe because all bridge functions that accept the handle are
documented as thread-safe in the header.

## SPM: Linking a Pre-Built Static Library

`Package.swift` cannot use `headerSearchPath` with absolute paths — it only
accepts paths relative to the target's source directory. Use `unsafeFlags` for
anything that requires absolute paths resolved at package-eval time:

```swift
let packageDir = URL(fileURLWithPath: #file).deletingLastPathComponent()

// Bridging header — must be absolute
.unsafeFlags(["-import-objc-header", packageDir.appendingPathComponent("bridge/include/magentart_bridge.h").path])

// Include path for the C++ library headers
.unsafeFlags(["-I\(mrt2BuildDir)/include"])

// Linker
.linkedLibrary("magentart_all"),
.unsafeFlags(["-L\(mrt2BuildDir)"])
```

## Merging Transitive Static Library Dependencies

A C++ library like MRT2 links against many transitive static deps (TFLite,
MLX, SentencePiece, abseil, ruy, …). SPM does not understand cmake's
transitive dependency graph, so you must provide all of them at link time.

The cleanest approach: use `libtool -static` to merge all transitive `.a`
files into one fat archive. The full list of required archives is in cmake's
generated `link.txt`:

```bash
BUILD=magenta-realtime/build
grep -oE '\.\./\.\./\.\./[^ ]+\.a' \
  "$BUILD/examples/mrt2/standalone/.../link.txt" \
  | sed "s|../../../|$BUILD/|g" \
  > /tmp/libs.txt
xargs libtool -static -o mrt2-build/libmagentart_all.a < /tmp/libs.txt
```

The merged archive is git-ignored and regenerated by `make build-mrt2`.

## `swift run` vs `swift build --product`

`--product` is a `swift build` flag only. `swift run` takes the executable
name as a positional argument:

```bash
swift build -c release --product magenta-player   # correct
swift run magenta-player                           # correct
swift run --product magenta-player                 # error: unknown option
```

## Deployment Target Must Match the Library

If the C++ library was compiled for macOS 14.0 (`-mmacosx-version-min=14.0`),
`Package.swift` must also target 14.0 — otherwise the linker emits warnings
and some symbols may behave unexpectedly. `.v14` requires PackageDescription
5.9; use the string form for 5.7 compatibility:

```swift
platforms: [.macOS("14.0")]
```
